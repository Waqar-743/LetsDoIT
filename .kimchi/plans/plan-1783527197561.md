# LetsDOiT — UI Design Plan

## Goal
Produce a clickable **HTML/CSS/JS prototype** of the LetsDOiT platform covering all 25 specified screens (auth, student, teacher) plus a companion **design-system markdown reference** that documents the visual language, components, and patterns used. The prototype must feel like a real SaaS education product, not a chatbot wrapper.

---

## Constraints
- **No build step, no frameworks.** Vanilla HTML + CSS + JS, runnable by opening `index.html` directly (file://) or via a simple static server.
- **Chart.js via CDN** for all charts (line, bar, doughnut, radar, area, flow).
- **Mocked AI conversations only.** Pre-canned sample messages, no streaming, no LLM calls.
- **All 25+ screens covered**, each functional with mock data, but minimal micro-animation.
- **Local mock data**, no backend. Student/teacher/role state held in `localStorage` so refreshes keep the role.
- **Privacy rules enforced in data layer**: students cannot see other students' names; teachers see only their own enrolled students.
- **Role-gated navigation**: switching role re-renders the sidebar/top bar.
- **Production-quality polish**, not a student project — but explicit "basic polish" scope (no fancy empty-state illustrations, no skeleton-loaders on every screen).

## Chunks

### Chunk 1 — Foundation: tokens, base styles, mock data, router shell
**Scope**
- `tokens.css`, `base.css`, `components.css`, `layout.css`, `screens.css`
- `data.js` (mock dataset — see "Mock Data Shape" below)
- `state.js` (session: `{role, user, aiMode, currentCourseId}`, persisted to `localStorage`)
- `router.js` (hash-based: `#/student/overview`, `#/teacher/courses`, `#/auth/welcome`, etc.)
- `shell.js` (renders sidebar + topbar based on role)
- `components.js` (`card()`, `badge()`, `kpiTile()`, `emptyState()`, `aiModeBadge()`, `progressRing()`, `avatar()`, `fileTypeIcon()`)
- `index.html` (loads Chart.js CDN, fonts, all CSS, all JS)

**Depends On**: nothing
**Accept When**
- Opening `index.html` shows the welcome screen with working sidebar/topbar logic after role-select.
- Switching role via topbar dropdown re-renders navigation without page reload.
- `localStorage` persists role across refresh.
- All chart placeholders render with Chart.js on a sample dataset.

**Open Questions**: None (resolved via questionnaire).

### Chunk 2 — Authentication flow (6 screens)
**Scope**
- `screens/auth.js` rendering:
  1. **Welcome** — product pitch, two big CTAs (Student / Teacher), mode-illustration
  2. **Role selection** — secondary confirmation, explains teacher-access restriction
  3. **Student signup** — name, email, password, dept, semester; "Create account"
  4. **Student login** — email + password, "Forgot password?", link to signup
  5. **Restricted teacher login** — email + password + "Teacher Access Key" field, banner explaining admin-only access, "Request access" link
  6. **Forgot password / recovery** — email field, success state with mock email-sent confirmation
- After successful student login → `#/student/overview`
- After successful teacher login (with valid mock access key) → `#/teacher/overview`

**Depends On**: Chunk 1
**Accept When**
- Each of the 6 auth screens is reachable via direct hash URL.
- Form validation produces inline error states on empty submit.
- "Continue as Student" and "Continue as Teacher" buttons visibly differ in weight/copy to signal restriction.
- Forgot-password shows a "check your email" success state.

### Chunk 3 — Student dashboard (11 screens)
**Scope**
`screens/student.js` rendering:

1. **Overview** (`#/student/overview`)
   - Greeting + preparation-level hero card (ring + label: *Good Preparation*)
   - 4 KPI tiles: Courses Joined, Quizzes Attempted, Avg Score, Study Streak
   - **Activity line chart** (last 14 days, sessions/minutes)
   - **Preparation-level radar** (5 dimensions: Notes Read, AI Practice, Quiz Accuracy, Weak-area Improvement, Consistency)
   - Enrolled courses (horizontal card list, 3–4 cards)
   - Recent AI conversations (last 3)
   - Upcoming quizzes/deadlines

2. **Join course** (`#/student/join`)
   - Big 4-digit code input (4 boxes, auto-advance), "Join" button
   - Mock validation: `1234` succeeds → "Joined Calculus 101 by Ms. Ayesha"; `0000` fails with "Invalid or expired code" toast
   - "How do I get a code?" help block

3. **My courses** (`#/student/courses`)
   - Grid of course cards (title, teacher name, enrolled count — e.g. "42 students enrolled", student-count never shows names), status pill (Active/Inactive)
   - Filter: All / Active / Archived
   - Per-card menu: Open / Leave course (with confirmation modal)

4. **Course detail** (`#/student/course/:id`)
   - Header: course title, teacher name, subject, code, enrolled count, preparation ring for *this course*
   - Tabs: **Material** | **Quizzes** | **AI Assistant** | **About**
   - Material tab: list of files with type icon, title, course, teacher, upload date, "View / Download / Send to AI" actions
   - Quizzes tab: list of available quizzes split into *Practice* (AI allowed) and *Test* (AI locked); each with due date, difficulty, time limit, "Start" button

5. **Material viewer** (`#/student/course/:id/material/:fileId`)
   - Two-pane: file preview placeholder (PDF iframe-style mock, slide thumbnails, note text) + right sidebar with AI actions ("Summarize", "Explain", "Generate practice questions", "Translate to Urdu")
   - Breadcrumb: Courses › Calculus 101 › Lecture 3 — Derivatives

6. **AI assistant** (`#/student/course/:id/ai` and global `#/student/ai`)
   - Full chat layout: message list, input box, AI-mode selector (Offline / Online / Hybrid) at top with brief explainer
   - Mode card shows current mode, latency indicator ("Local Gemma • 2.3 GB loaded"), and a one-line description per mode
   - Suggested prompts (chip row): "Explain last lecture", "Generate 5 MCQs from notes", "Find my weak areas", "Make a study plan"
   - Mock conversation thread demonstrating structured AI output (headings, bullets, math block, quiz block)
   - If user is on a Test-mode quiz, banner explains "AI is disabled during this test"

7. **Quiz setup** (`#/student/course/:id/quiz/new`)
   - Difficulty selector: Easy / Medium / Hard (segmented control)
   - Question count slider (5–25)
   - Type multi-select: MCQ, Short, Conceptual, True/False, Mixed
   - Source: which materials to draw from (checkboxes)
   - "Practice Mode" toggle (default ON — AI help allowed)
   - "Generate Quiz" → loading state → redirects to attempt

8. **Quiz attempt** (`#/student/course/:id/quiz/:quizId/attempt`)
   - Question header (Q3 of 10), progress bar, timer (if timed)
   - Question card with answer UI per type
   - Footer: Previous / Mark for review / Next
   - Right rail: question navigator (grid of numbers), flagged indicator
   - "Submit" → confirmation modal
   - If Test mode: AI chat panel is **hidden**, sidebar AI link disabled, banner reads "Test Mode — AI disabled"

9. **Quiz result** (`#/student/course/:id/quiz/:quizId/result`)
   - Score hero (e.g. 7/10 = 70%) with grade label
   - Tabbed breakdown:
     - **Correct answers** (expandable Q&A)
     - **Mistakes** with mistake-type tag: *Conceptual gap / Careless mistake / Misread question / Partial understanding*
     - **Weak topics** (chips, each linked to relevant material)
     - **Suggested practice** (auto-generated re-quiz CTA + material links)
     - **Preparation level update** (delta: ↑ from Average to Good)
   - Bar chart: per-question accuracy

10. **Preparation level** (`#/student/preparation`)
    - Hero ring with current label
    - Trend chart (preparation over time)
    - Per-course preparation cards
    - Definition block explaining how the level is calculated

11. **Profile/Settings** (`#/student/profile`)
    - Avatar (upload mock), name, email, dept, semester
    - Joined courses list (same as #3 but with manage actions)
    - Preferences: AI mode default, notifications, language (English / Urdu)
    - Logout

**Depends On**: Chunk 1
**Accept When**
- All 11 student screens reachable by direct URL.
- Charts (activity, radar, accuracy) render with mock data.
- Privacy rule enforced: course views never expose individual student names; only aggregated "N students enrolled".
- Test-mode banner correctly disables AI assistant link in nav.

### Chunk 4 — Teacher dashboard (12 screens)
**Scope**
`screens/teacher.js` rendering:

1. **Overview** (`#/teacher/overview`)
   - Greeting + KPI row: Active Courses, Total Students, Active This Week, Avg Class Score
   - **Class activity heatmap** (Chart.js matrix-style) + bar chart (quiz attempts per day)
   - **Course engagement table**: course, enrolled, active this week, avg score, risk count
   - **Students at risk** (sorted list with risk badge: Low/Med/High)
   - **Class-wide weak topics** (horizontal bar)
   - **Teacher workflow flow chart**: Course Created → Material Uploaded → Quiz Generated → Students Attempted → Weak Areas Detected → Teacher Reviews

2. **Create course** (`#/teacher/courses/new`)
   - Form: title, description, subject, class/semester, optional image upload
   - Course-code preview (4-digit, auto-generated, "Regenerate" button, "Copy" button)
   - Status toggle: Active/Inactive
   - "Create course" → success state showing code → redirect to course manage

3. **Course management** (`#/teacher/course/:id`)
   - Same header as student view but with **Teacher** actions: edit, archive, regenerate code, view code
   - Tabs: **Students** | **Materials** | **Quizzes** | **Analytics** | **Settings**
   - Students tab: list of enrolled students (avatar + name + last active + preparation level + risk), search/filter, "Remove student" with confirmation
   - Materials tab: file list + "Upload" CTA, per-file actions (preview, download, delete, organize)
   - Quizzes tab: list of quizzes with status badges (Draft / Published / Active / Closed), per-quiz menu
   - Analytics tab: per-course analytics (see #10 below)
   - Settings tab: edit course fields, archive

4. **Upload material** (`#/teacher/course/:id/upload`)
   - Drag-and-drop zone (multiple files)
   - Per-file row: thumbnail, filename, type (auto-detected), lecture/topic/week selector, "Save" button
   - Bulk upload supported; queue with progress

5. **Generate quiz** (`#/teacher/course/:id/quiz/new`)
   - Quiz title, source material checkboxes, difficulty, # questions, question types, time limit, due date
   - **Mode toggle**: *Practice quiz* (AI allowed for students) vs *Official test* (AI disabled, with integrity warning)
   - Publish state: Draft / Publish now / Schedule
   - "Generate with Gemma" → AI thinking state (skeleton + "Generating questions from 4 documents…") → redirects to review

6. **Review/edit quiz** (`#/teacher/course/:id/quiz/:quizId/review`)
   - Generated questions list, each editable inline (stem, options, correct answer, explanation)
   - "Add question", "Re-generate this question", "Delete"
   - Right rail: rubric settings, marks-per-question
   - Footer: "Save draft" / "Publish"

7. **Publish quiz/test** (`#/teacher/course/:id/quiz/:quizId/publish`)
   - Pre-publish checklist (questions reviewed, due date set, mode set, etc.)
   - "Publish to students" with confirmation modal that explicitly restates Practice vs Test mode behavior

8. **Student list** (`#/teacher/students`) — across all courses
   - Table: student name, courses enrolled, avg score, prep level, last active, risk
   - Sort, filter by course, search
   - Click row → student detail

9. **Student detail/activity** (`#/teacher/student/:id`)
   - Profile header, joined courses
   - **Activity timeline** (Chart.js line: practice minutes per day)
   - Quiz attempts table (date, quiz, score, mistakes)
   - Weak areas (chips with mistake-type breakdown)
   - Preparation level per course (small ring + label)
   - AI usage activity (sessions, topics)
   - "Message student" (mock), "Remove from course"

10. **Class analytics** (`#/teacher/course/:id/analytics` and global `#/teacher/analytics`)
    - Score distribution histogram
    - Topic mastery heatmap (topic × student bucket)
    - Preparation-level distribution (doughnut)
    - Common misconceptions list with frequency
    - Engagement: materials viewed, AI sessions, quiz attempts
    - Export CSV (mock)

11. **Teacher AI assistant** (`#/teacher/ai`)
    - Same shell as student AI, but with **teacher-flavored prompts**: "Summarize last lecture", "Generate 10 MCQs from notes", "Find class-wide weak areas", "Create Urdu-English explanation", "Draft answer key"
    - AI mode selector identical to student (Offline/Online/Hybrid)

12. **Teacher profile/settings** (`#/teacher/profile`)
    - Avatar, name, email, subjects taught
    - Teacher access status banner ("Verified by admin")
    - Notifications, AI default mode
    - Logout

**Depends On**: Chunk 1
**Accept When**
- All 12 teacher screens reachable.
- Generate-quiz "AI thinking" state shows for 1.5 s, then transitions to review screen.
- Workflow flow chart on teacher overview renders as a 6-step horizontal flow (custom SVG inside Chart.js or static SVG).
- Students tab only shows students enrolled in *that* teacher's course (privacy rule).

### Chunk 5 — Cross-cutting polish & design-system docs
**Scope**
- `design-system.md` covering:
  - **Brand**: name, voice (calm, professional, education-first, Urdu/English bilingual copy)
  - **Color tokens**: primary (teal/indigo for trust), accent (warm amber for highlights), neutrals, semantic (success/warning/danger/info), AI-mode colors (Offline=slate, Online=blue, Hybrid=violet)
  - **Typography**: Inter for UI, JetBrains Mono for code/math, type scale
  - **Spacing, radius, shadow, motion**
  - **Component inventory**: Button, Card, Badge, Input, Modal, Toast, Tabs, Avatar, File-type icon, AI-mode chip, Preparation ring, Risk badge, Empty state, Loading skeleton, Error state
  - **Screen patterns**: dashboard layout, course page layout, AI chat layout, quiz layout, modal patterns
  - **Copy guidelines**: privacy phrasing ("We never share your name with classmates"), AI-mode microcopy, practice-vs-test microcopy
- `README.md`: how to open, hash-route map, what's mocked, keyboard shortcuts (Esc closes modals, g+s goes to student overview, g+t to teacher overview)
- Responsive checks: collapse sidebar to icon rail at <1024px, mobile drawer at <768px (basic, not exhaustive)
- Empty/loading/error states on every list view (no half-finished screens)
- Final cross-link pass: every CTA navigates somewhere

**Depends On**: Chunks 1–4
**Accept When**
- Every screen reachable from the sidebar in ≤2 clicks from role landing.
- Empty state visible on filtered list with zero results.
- Privacy rule verified: `grep` for `student.name` inside any student-side render path returns no matches against other students.
- `design-system.md` opens and lists every component actually used in CSS.
- Tab through the whole prototype: no console errors, no broken images, no 404s.

## Verification Strategy

| Check | How |
|---|---|
| All screens reachable | Manually traverse hash-route map; checklist in `README.md`. |
| Privacy rule | Code review: `data.js` exposes per-student data only via teacher-side accessors; `screens/student.js` never imports `students[id].name` except for own profile. |
| Charts render | Visual confirmation on overview, prep-level, quiz-result, teacher-analytics screens. |
| AI-mode selector works | Clicking each mode updates `state.aiMode` and `aiModeBadge` everywhere. |
| Test-mode disables AI | `state.activeQuizMode === 'test'` → `sidebar.js` hides AI link, AI page renders banner. |
| Role switching | Topbar role-switcher flips sidebar/topbar without reload. |
| localStorage persistence | Refresh on any screen restores role + last route. |
| Responsive | Resize to 1280, 1024, 768, 480 — sidebar collapses, no horizontal scroll. |
| No console errors | Open DevTools, click through every screen. |
| Design-system accuracy | Every CSS class used has an entry in `design-system.md`. |

## Mock Data Shape (summary)

```js
{
  students: [{id, name, email, dept, semester, avatar, joinedCourses, prepLevel, risk}],
  teachers: [{id, name, email, subjects, verified, avatar}],
  courses:  [{id, title, description, subject, code, teacherId, studentIds, status, createdAt}],
  materials:[{id, courseId, type, title, uploadDate, size, topic, week}],
  quizzes:  [{id, courseId, title, mode:'practice'|'test', difficulty, questions:[...], timeLimit, dueDate, published, scores:[{studentId, score, mistakes}]}],
  activity: [{studentId, date, minutes, sessions, aiPrompts}],
  aiChats:  [{userId, role, messages:[{role:'user'|'ai', content, mode}]}],
  prepLevels: {needs_practice, average, good, excellent, outstanding}
}
```

All names use culturally appropriate Pakistani names; courses use realistic Pakistani curriculum subjects (Calculus, Pakistan Studies, Urdu, English, Computer Science, Physics).

## Decision Log

| Decision | Choice | Rejected | Rationale |
|---|---|---|---|
| Deliverable | HTML prototype + design-system docs | React app, spec-only | User-selected; balances tangibility with delivery speed. |
| Charts | Chart.js CDN | Custom SVG, ApexCharts | User-selected; simple, no build. |
| AI chat | Mocked only | Simulated streaming, real LLM | User-selected; lowest complexity. |
| Scope | All 25+ screens, basic polish | Core-flows only | User-selected. |
| Routing | Hash-based SPA | Multi-file HTML | Single-file easier to navigate in plan; one shared CSS. |
| State | `localStorage` + in-memory module | No persistence | Refresh-friendly demo without backend. |
| Framework | Vanilla JS | React/Vue/Svelte | No-build constraint. |
| Iconography | Inline SVG sprite | Icon font, image files | Crisp at any size, one HTTP fetch, easy to recolor. |
| Fonts | Inter + JetBrains Mono via Google Fonts | System fonts only | More product-feel; still no build step. |
| Privacy enforcement | Data-layer accessors | UI-layer filtering only | Defense-in-depth — easier to audit. |

## Risks

| Risk | Likelihood | Mitigation |
|---|---|---|
| 25 screens in one pass → inconsistent visual language | Medium | Design tokens in Chunk 1 used everywhere; `design-system.md` is the source of truth, written alongside components. |
| Chart.js doesn't natively support flow charts / heatmaps | Medium | Use Chart.js for line/bar/doughnut/radar; build flow chart and heatmap as static SVG inside `components.js`. |
| File:// CORS blocks Chart.js CDN or Google Fonts | Low (modern browsers allow) | README documents `python -m http.server` as fallback; if needed, vendor Chart.js locally. |
| Privacy rule accidentally leaks | Medium | Single `data.js` exposes teacher-only accessors named `teacherStudents(teacherId)`; student-side renderers never call them. |
| Scope creep into animations/transitions | Medium | "Basic polish" explicitly excludes micro-animation; only essential state transitions (loading → loaded, empty → populated). |
| Quiz generation "AI thinking" feels too quick | Low | Hard-code 1500 ms delay with skeleton state in Chunk 4. |
| Responsive breaks on very small screens | Low | Mobile is out-of-scope per spec ("desktop and web"); provide a graceful single-column at <768px but don't optimize further. |

---

## Out of Scope (explicit)
- Real authentication / backend / database
- Real LLM calls (Gemma integration)
- Mobile-native or PWA installability
- Internationalization beyond a single Urdu/English toggle in profile
- Payment, notifications, email sending
- Accessibility audit beyond basic focus rings and semantic HTML
- Unit/integration tests (verification is manual per checklist)

---

This plan covers every screen called out in the spec, encodes the privacy and teacher-restriction rules in the data layer, and produces both an interactive prototype and the design documentation needed to hand the system off to engineers.
